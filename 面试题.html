一、架构设计

1. 为什么选择纯前端 SPA 而不加后端？什么场景下需要加后端？

选纯前端的原因：

  • 作业要求"面试官能在本地运行"，每多一个依赖（Python、数据库）就多一个跑不起来的风险
  • 所有素材是静态文件（24 个），数据量小，前端直接 fetch cases.json 即可
  • 3D 渲染依赖 WebGL，天然在前端完成
  • 标注数据用 IndexedDB 持久化，不需要后端数据库

需要加后端的场景：

  • 多用户协作：多人同时评审同一批素材，需要后端同步状态
  • 素材量放大：几千上万个素材，需要用后端做分页、搜索、权限控制
  • 接真实 AI API：API Key 不能放前端，需要后端代理
  • 版本管理：素材的多版本迭代、操作日志
2. Zustand 在这个项目里扮演什么角色？为什么不用 Redux？

Zustand 管理的全局状态：

  • assets / filteredAssets：全部素材和筛选后的列表
  • filter：当前的搜索/状态/分类/类型筛选条件
  • viewMode：grid / detail / compare 三种视图
  • selectedAssetId / compareList：选中和对比列表
  • 各个素材字段的更新函数（status/notes/tags/aiSummary）

为什么不用 Redux：Redux 在这个规模下太重了。Zustand 只有 0.8KB，不需要 Provider 包裹、不需要 action creator / reducer 模板代码。对于 5 个状态字段 + 12 个 action 的规模，Zustand 刚刚好。Redux Toolkit 更适合多模块、有中间件需求的复杂应用。

3. 筛选逻辑放在前端还是后端？为什么？

放在前端，在 Zustand 的 applyFilter() 函数里做纯数组过滤。

理由：

  • 总共 24 个素材，JS 内存里做 filter 比 HTTP 请求快几个数量级
  • 不需要服务端分页
  • 筛选是纯计算（输入数组→输出数组），无副作用，纯前端做比后端简单

什么情况下需要移到后端：素材量超过 1000 条时，前端一次性加载 + 筛选会有性能压力，此时应该后端做分页查询。

4. Segmented 组件切换视图的校验逻辑是怎样设计的？

Toolbar.tsx 中 handleViewModeChange 做了两层校验：

  • 切到「详情」视图：检查是否有 selectedAssetId，没有则弹出提示 请先在网格中点击一个素材卡片查看详情，阻止切换
  • 切到「对比」视图：检查 compareList 是否为空，为空则弹出 请先在网格中勾选要对比的素材，阻止切换

为什么是 Segmented 的 onChange 拦截而不是在 App.tsx 里兜底渲染：在 Segmented 层面就直接阻止了，视觉上 Segmented 不会跳过去再弹回来，交互更自然。App.tsx 里如果 viewMode 到了但条件不满足会空白页，那是被动修复。

二、3D 渲染

5. GLB 和 STL 两种格式在加载和渲染上有什么本质区别？

GLB（glTF Binary）：

  • 自带材质（PBR 参数）、纹理贴图、骨骼动画、场景层级
  • 使用 drei 的 useGLTF() 加载，返回完整的 scene 对象
  • 渲染时材质直接生效，模型自带颜色和质感
  • 文件通常含多张纹理贴图

STL：

  • 只有三角网格几何体，无材质、无纹理、无颜色
  • 使用 STLLoader 加载，返回 BufferGeometry
  • 需要手动创建 mesh + meshStandardMaterial，指定颜色
  • 适合医学 CT 重建模型（只有几何轮廓，不需要纹理）

实现上的统一：ModelViewer 对外暴露统一的 Props（file + format），内部根据 format 判断走哪条加载路径。STL 模型统一用灰色 #b0bccd + 漫反射材质。

6. useAutoScale 的实现原理？为什么要用 requestAnimationFrame？

原理：

function useAutoScale(ref, deps) {
  useEffect(() => {
    requestAnimationFrame(() => {
      const box = new THREE.Box3().setFromObject(ref.current)
      const size = box.getSize(new THREE.Vector3())
      const maxDim = Math.max(size.x, size.y, size.z)
      ref.current.scale.setScalar(2.5 / maxDim)
    })
  }, deps)
}

为什么用 requestAnimationFrame：GLB 模型加载是异步的，useEffect 触发时 <Group> 和 <primitive> 已经在 DOM 中，但子节点（模型的 mesh）可能还没有挂载到 scene graph 上。rAF 保证在下一帧执行，此时 scene graph 已完整，Box3.setFromObject 才能遍历到所有子节点。

为什么缩放到 2.5：默认 camera 位置是 [3, 2, 5]，2.5 单位半径的球体在这个 camera 距离下刚好占画面 60-70%，既不会太大撑满屏幕，也不会太小看不清。

7. 为什么移除了 @react-three/drei 的 Stage 组件，改用纯手动灯光？

项目早期用 Stage 组件，它自动加载 HDR 环境贴图 + 自动曝光 + 接触阴影。但在实际运行中发现：

  • 环境贴图加载失败会导致整个 Canvas 白屏，Stage 内部没有暴露加载错误的回调
  • Stage 的阴影需要额外的渲染开销，对素材预览来说不需要
  • R3F 的错误边界只打 warning 不抛异常，Stage 内部任何组件加载失败，Canvas 直接白屏，控制台无报错

替换为纯手动灯光后（ambientLight × 1 + directionalLight × 2 + hemisphereLight × 1），所有灯光都是 Three.js 内置的基础组件，零外部资源依赖，不可能因加载失败而白屏。

8. STL 几何体加载后为什么要 clone()？不 clone 会有什么问题？

useLoader(STLLoader, file) 会按 URL 缓存 BufferGeometry 对象。如果不 clone：

// 错误做法：
const geometry = useLoader(STLLoader, file)
geometry.translate(x, y, z)  // 直接修改了缓存对象！

第一次加载 → 居中正确。第二次加载 → 因为 URL 没变，useLoader 返回缓存对象，但对象已经被 translate 过一次了，再次使用 useMemo 传入的 geometry 引用相同，不会重算。但如果有另一个场景（如对比视图两个画布同时加载），两个 Canvas 共用同一个 Geometry 实例，互相 contamination。

正确做法：

const rawGeometry = useLoader(STLLoader, file)
const geometry = useMemo(() => {
  const clone = rawGeometry.clone()
  clone.translate(x, y, z)
  clone.scale(s, s, s)
  return clone
}, [file])  // 依赖 file，切文件时重新 clone 和居中

三、状态与交互

9. 为什么点击 Segmented 的「详情」时没有反应？你是怎么修复的？

根因：App.tsx 中的渲染条件是 viewMode === 'detail' && selectedAssetId && <DetailView />。直接点「详情」时 selectedAssetId 为 null，条件短路，渲染空 Fragment，页面空白且没有任何提示。

修复：给 Segmented 的 onChange 加前置校验：

if (mode === 'detail' && !state.selectedAssetId) {
  message.info('请先在网格中点击一个素材卡片查看详情')
  return  // 阻止切换，Segmented 不动
}

为什么不直接在 App.tsx 里渲染提示组件：在 Segmented 层面拦截更自然——按钮按了但不动，+ 弹出引导提示。用户知道"这个操作现在还不能做"，而不是被跳到一个空白页。

10. 对比视图的交互流程是怎样的？边界情况如何处理？

正常流程：

  1. 网格列表中勾选素材卡片右下角的复选框（最多 2 个）
  2. 勾满 2 个时自动跳转到对比视图
  3. 对比视图左右分栏展示两个素材，注意对比视图中也有「返回列表」按钮
  4. 点击「返回列表」→ clearCompare + setViewMode('grid')

边界情况：

  • 未勾选任何素材直接点「对比」→ Segmented 拦截 + 提示"请先勾选素材"
  • 只勾选 1 个→ 对比视图左侧显示素材，右侧虚线框 + "再选一个素材进行对比"
  • 对比视图中勾选的素材被删除→ filter(Boolean) 保证不会渲染 undefined
  • 空状态→ 显示 Empty 组件 + 引导文案 + 「返回网格列表」按钮
11. 筛选搜索的 applyFilter 为什么设计在 store 里而不是组件里？

因为筛选结果 filteredAssets 是全局状态，多个组件都需要用到：

  • AssetGrid：渲染筛选后的素材列表
  • Header：显示"待评审"数量（从 assets 算，不是 filteredAssets）
  • 未来可能加的 统计面板：也要基于当前筛选结果做统计

放在 store 里，filter 变化时自动更新 filteredAssets，所有消费 filteredAssets 的组件一次 re-render。如果放在某个组件里，其他组件拿不到筛选结果，或者需要 props 逐层传递。

setFilter: (partial) => {
  const filter = { ...get().filter, ...partial }
  set({ filter, filteredAssets: applyFilter(get().assets, filter) })
}

四、AI Mock 设计

12. 为什么用 Mock 而不是真实 API？如果换成真实 API 需要改什么？

用 Mock 的原因：

  • 面试题明确要求"真实 AI 接入不是必须"
  • 真实 API 需要 API Key，面试官本地运行时需要额外配置，徒增失败点
  • Mock 覆盖了完整的交互流程（加载态 → 结果展示 → 采纳交互），功能演示目的已达到

换真实 API 要改的地方：

只改 src/services/aiMock.ts 一个文件。三个导出函数的签名不变：

// Mock 版本
export async function mockAIAnalyze(..., existingTags) → { summary, suggestedTags }

// 真实 API 版本（只需改函数体，不改签名）
export async function mockAIAnalyze(..., existingTags) {
  const res = await fetch('https://api.deepseek.com/v1/chat/completions', ...)
  const data = await res.json()
  return { summary: data.choices[0].message.content, suggestedTags: parseTags(data) }
}

调用方 AnnotationPanel 不需要改一行代码。

13. Mock 的摘要生成策略是什么？有什么局限？

策略：模板字符串拼接。根据素材类型（image/model3d）选择不同的模板池，然后用随机插入的方式填充占位符。例如：

// 图片模板
"这是一张{category}类素材，画面{mood}，主要展示{subject}，适合用于{usage}场景。"

// 3D 模型模板
"这是一个{format}格式的{category}3D模型，{detail}，{quality}"

占位符被替换为：mood → randomItem(["明亮柔和", "色调偏暗", ...])，usage → randomItem(["设计参考", "素材库归档", ...])。

局限：

  • 摘要内容与素材实际内容无关——只是模板句子，不是语义理解
  • 同样的素材每次分析结果不同（random）
  • 标签推荐只是从池中随机抽取，不是真正的语义标签
14. AI 推荐标签如何"一键采纳"？

AnnotationPanel 渲染 AI 推荐标签时，每个标签绑定了 onClick：

{asset.aiSuggestedTags.map(tag => (
  <Tag key={tag} style={{ cursor: 'pointer' }}
    onClick={() => {
      if (!asset.tags.includes(tag)) {
        updateTags(asset.id, [...asset.tags, tag])
      }
    }}
  >
    + {tag}
  </Tag>
))}

点击 AI 标签 → 检查是否已存在 → 调用 updateTags 将标签添加到该素材的 tags 数组。Zustand 的 tags 更新会同步到 store，所有消费该素材 tags 的组件（AssetCard、DetailView）自动 re-render。

为什么不自动全部采纳：让用户手动选择，符合审核场景中"AI 只是辅助，决策权在人"的原则。

五、数据与素材

15. cases.json 承担了什么角色？换成 1000 个素材需要改什么？

角色：cases.json 是项目的数据驱动核心——它是一个完整的素材元数据库。前端的 AssetCard/DetailView/CompareView 全部基于 cases.json 的字段来渲染，筛选逻辑基于这些字段做过滤。换一批素材只需要替换文件 + 更新 JSON。

换成 1000 个素材需要改：

  • 需要改的：在前端引入虚拟滚动（如 react-window），否则 1000 张卡片渲染会卡顿
  • 建议改的：筛选逻辑从纯前端改为后端分页查询（加一个简单的 FastAPI 或 json-server）
  • 不需要改的：所有 UI 组件、状态管理、3D 查看器、AI Mock 都不需要改
16. 素材文件放在 public/ 目录下而不是 src/ 里，为什么？

Vite 的 public/ 目录下的文件会被原样复制到构建输出目录，不经过 Webpack/Vite 的模块打包。素材文件（图片、GLB、STL）是体积很大的二进制文件：

  • 如果放在 src/ 里通过 import 引入，Webpack 会尝试处理它们（生成 hash 文件名、优化路径），对于几十 MB 的 GLB/STL 文件来说纯属浪费构建时间
  • public/ 目录下的文件在运行时通过 /sample-assets/xxx 直接访问，路径透明，调试方便
  • 素材不参与模块依赖图，修改素材也不需要触发 HMR 重新编译

访问路径规则：public/sample-assets/images/a.jpg → 运行时 /sample-assets/images/a.jpg

17. 24 个素材的初始状态（pending/approved/rejected）是怎么分配的？为什么？

分配策略：

  • pending（待评审）：16 个，占大多数，模拟未开始的审核队列
  • approved（已通过）：5 个，模拟已经审过的优质素材
  • rejected（已拒绝）：3 个（tech_workspace、ToyCar、mail-006），模型中各有一个 rejected

为什么这样分配：让三种状态在筛选器中都有数据可展示，且大部分是"未处理"状态符合真实审核场景的初始状态分布。优先级 1/2/3 也覆盖了所有三个等级，方便测试按优先级排序功能。

六、Bug 修复经验

18. 网格缩略图不显示的原因和修复方案？

原因：AssetCard 用 CSS background: url(${asset.file}) 加载图片，asset.file 的值是 images/xxx.jpg（相对路径），缺少 /sample-assets/ 前缀，浏览器请求变成 http://localhost:3000/images/xxx.jpg（404）。CSS background 加载失败时静默失败，无 console error，只显示灰色底。

修复：

  1. 新增 getThumbnailUrl() 拼接完整路径：/sample-assets/${asset.file}
  2. 把 CSS background 替换为 <img> 标签 + onError 兜底：
// before
<div style={{ background: `url(${asset.file}) ...` }} />

// after
<img src={getThumbnailUrl(asset)} onError={() => setImgError(true)} />

使用 <img> 的好处：加载失败会触发 onError，可以换回占位图标，而不是无提示地空白。

19. 3D 模型白屏的原因和修复方案？

原因：两重问题叠加。

第一重（白屏）：使用 <Stage> 组件，内部加载 HDR 环境贴图。网络/缓存问题导致加载失败时，R3F 的 ErrorBoundary 只会打 console.warning 不会抛出异常，Canvas 直接白屏，用户无法看到任何错误信息。

第二重（加载转圈）：R3F 的 Suspense 包裹的组件如果在加载中抛出异常，Suspense 的 fallback 会一直显示，无法退出。

修复：

  1. 移除 Stage，改用纯手动灯光（ambient + 2× directional + hemisphere），零外部资源依赖
  2. 引入 LoadingMonitor 组件，用 drei 的 useProgress 监听加载错误
  3. 检测到错误时展示 Alert + 重试按钮，通过 key 重置 Canvas
20. 模型名称不匹配（脑干→人形角色）是怎么发现的？

Khronos 的 BrainStem.glb 文件来自 glTF-Sample-Assets 仓库，该仓库的所有模型都是 glTF 标准测试用例，BrainStem 是一个带骨骼动画的人形角色测试模型（用来测试 skinning/joint 功能）。我最初根据文件名中的 "BrainStem" 望文生义，认为是医学脑干解剖模型。

修复：重新核查了 Khronos 仓库中该模型的说明文档，确认它是角色测试模型而非医学模型。在 cases.json 中将名称改为"人物角色 - BrainStem"，分类从"生物医学"改为"角色"，标签和描述同步修正。

教训：为第三方素材命名时不能只看文件名，应该先加载预览确认内容后再写名称和描述。

21. 为什么模型加载后有的太大有的太小？怎么统一适配的？

原因：不同的 GLB 模型原始尺寸差异巨大——DamagedHelmet 约 0.3 米、CesiumMan 约 1.7 米、AntiqueCamera 约 0.5 米。固定 camera 位置下，小模型只剩一个点，大模型撑满屏幕。

修复：新增 useAutoScale hook：

  1. 用 Box3.setFromObject(group) 计算模型的包围盒
  2. box.getSize() 得到模型实际尺寸
  3. 计算缩放比例 2.5 / maxDim
  4. 通过 group.scale.setScalar() 统一缩放到 2.5 单位

对于 STL 模型，在 useMemo 中对 clone 后的 geometry 做相同的 scale 操作。

为什么是 2.5 不是 1：camera 距离 model 约 3-5 单位，2.5 半径的球体在这个距离下占画面 60-70%，视觉上舒适。如果缩放到 1，小模型在画面中会显小。

七、扩展与思考

22. 如果要支持更多 3D 格式（如 FBX、OBJ），需要改什么？

需要改 ModelViewer.tsx 一个文件：

function ModelScene({ file, format }) {
  return (
    <>
      <Suspense fallback={null}>
        {format === 'glb'  && <GLBModel file={file} />}
        {format === 'stl'  && <STLModel file={file} />}
        {format === 'obj'  && <OBJModel file={file} />}
        {format === 'fbx'  && <FBXModel file={file} />}
      </Suspense>
    </>
  )
}

新增对应的加载组件：

  • OBJ → three-stdlib 的 OBJLoader，与 STL 类似需要手动加 Material
  • FBX → three-stdlib 的 FBXLoader，返回 Group，与 GLB 类似

case.json 中新增素材时 format 字段设为对应值即可。不需要改动其他组件。

23. 如果评审标注需要多用户共享，架构上怎么改？

需要从前端单机架构改为 前端 + 后端 + 数据库 架构：

  1. 后端：加一个轻量 API 层（FastAPI 或 Next.js API Routes）
  2. 数据库：用 PostgreSQL 或 SQLite（单机小团队场景足够），保存 评审记录表（asset_id, reviewer_id, status, notes, tags, updated_at）
  3. 认证：简单 JWT 或 session 认证，区分不同评审人
  4. 实时同步：如果两个评审人同时标注同一素材，用 WebSocket（Socket.IO）或轮询同步状态变更
  5. 冲突处理：简单策略"最后保存者获胜"，复杂策略用版本号乐观锁

现有的 Zustand store 可以留着做前端缓存，每次操作先更新本地状态，再向后端发同步请求。

24. 这个项目哪些地方体现了 AI 在研发中的使用？（作业硬性要求）

三个方面：

  1. AI Mock 辅助评审（产品功能）：详情页中的「AI 分析」按钮，一键生成素材摘要 + 推荐标签，用户可点击采纳。模拟了 AI 辅助素材审核的场景，即使使用 Mock 也展示了完整的交互流程
  2. 本项目的代码实现由 AI（Claude）辅助完成：从框架脚手架搭建、组件代码生成、Bug 修复到文档编写，全程与 AI 协作完成。AI 使用说明文档中记录了"如果替换为真实 API 需要改什么"，展示了理解而非黑盒使用
  3. 用户自行探索：项目中还可以尝试用 AI 生成更多测试素材、用 AI 自动生成 cases.json 的素材描述，这些方向在 AI 使用说明中也有提及
25. 如果让你重构或优化这个项目，你会优先做什么？

优先级排序：

  • P0 - 素材加载体验：大 GLB 文件（7MB+）加载时间较长，加一个进度条或 Loading 骨架屏，而不是纯 Spin
  • P0 - 标注持久化：当前的 status/notes/tags 只存在 Zustand 内存中，刷新就丢。接入 IndexedDB（idb-keyval 已安装），组件卸载/页面刷新时自动持久化
  • P1 - 图片懒加载：24 个素材不多，但如果扩到几百个，需要虚拟滚动 + Image lazy loading
  • P1 - 3D 模型预加载：首页网格加载时提前预加载 GLB 文件，减少进入详情页的等待时间
  • P2 - Code Splitting：Three.js + drei + three-stdlib 占 1.7MB（gzip 后 500KB），可以做 lazy load 只在进入详情页时加载
  • P2 - 键盘快捷键：左右键切换素材、A/R/P 快速标记状态、Ctrl+Enter 提交评审等
